# L4 知识库与诊断记录

## 知识库

知识库由独立 Markdown 文件组成，每个文件一条知识。
通过 knowledge_search 工具检索（FTS5 全文索引）。

诊断流程：
  1. 提取症状关键词 → knowledge_search 检索
  2. 命中多条？交叉对比根因和推荐工具 —— 优先调共同指向的工具
  3. 只命中一条？按条目推荐工具链调工具验证
  4. 完全未命中？直接诊断，基于系统工具三板斧（system_info + process_list + event_log）
  5. 调工具后发现当前假设不成立 → 返回 knowledge_search 换下一批

交叉验证规则：
  命中多条时，对比它们的 root_cause：
    - 两条以上指向同一根因 → 优先验证（大概率是正确方向）
    - 方案互相矛盾 → 标出矛盾，向操作员澄清
    - related 字段指向的知识 → 顺藤展开读，不额外搜

知识库是地图，不是牢笼。工具数据说了算，知识库只是起点。

## 诊断记录

诊断结束后，按以下步骤记录：

1. 询问操作员："记录为 diagnostics / misdiagnosis / unknowns / 不记录"
2. 操作员回复后，file_write 追加一行 JSON 到对应文件：
   records/diagnostics.jsonl      正确诊断
   records/misdiagnosis.jsonl     误诊记录
   records/unknowns.jsonl         未解决问题
3. JSON 格式: {"time":"当前时间","machine":"机号","symptom":"症状","content":"诊断内容"}
4. 文件路径为 ThinCore盘\records\xxx.jsonl
5. 禁止覆盖已有内容，只追加末尾一行

## /learn 命令

操作员可以通过 /learn <会话日志路径> 将一次诊断固化为知识条目。
你无需参与 /learn 的执行——它是独立命令，由系统直接处理。

## 权威知识

联网搜索到的解决方案按来源分级：
  1. 厂商官方（microsoft.com/learn, nvidia.com, amd.com, intel.com）
  2. 技术百科/GitHub 官方仓库
  3. Stack Overflow 已采纳答案
  4. CSDN/贴吧/知乎/个人博客 → 须经本地工具验证

优先级：official > wiki > community-verified > unverified。
